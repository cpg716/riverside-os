# Riverside OS v0.85.5 — 30-Minute Quick-Start Deployment Guide

**Goal: Get fully up and running in 30 minutes (sync time not counted)**

This guide takes you from zero to processing your first sale on Riverside OS. Print this guide and check off each step as you complete it.

---

## 30-Minute Timeline Overview

| Phase | Time | What You'll Do |
|-------|------|----------------|
| **Preparation** | 0-5 min | Download files, gather info, prep network |
| **Server Install** | 5-12 min | Install Backoffice/Server PC and verify |
| **Integrations** | 12-18 min | Configure critical integrations (QBO, Helcim, etc.) |
| **Register #1** | 18-22 min | Install main register with peripherals |
| **Register #2** | 22-25 min | Set up iPad PWA satellite lane |
| **Test Sale** | 25-30 min | Run smoke test and process first sale |

*Note: Initial data sync (Counterpoint, catalog, etc.) runs in background and doesn't count toward the 30 minutes.*

---

## Phase 1: Preparation (0-5 minutes)

### What You Need Before Starting

**Download Required Files:**
- [ ] Download from: https://github.com/cpg716/riverside-os/releases/tag/v0.85.5
- [ ] `RiversideOS-v0.85.5-Windows-Deployment.zip`
- [ ] `Riverside.POS_0.85.5_x64_en-US.msi`
- [ ] Unzip the deployment package

**Gather This Information:**
- [ ] Backoffice/Server PC name and LAN IP (e.g., `10.64.70.196`)
- [ ] Epson receipt printer IP or Windows printer name
- [ ] Scanner is in HID keyboard mode
- [ ] Payment terminal device codes (if using Helcim)
- [ ] Windows Administrator access on Server PC and Register #1

**Network Setup:**
- [ ] All devices (Server, Register #1, iPad) on same store network
- [ ] Confirm devices can reach each other
- [ ] Reserve static/DHCP IPs for server and printer (recommended)

**Store Roles (Quick Reference):**
| Station | Device | API Host |
|---------|--------|----------|
| Backoffice/Server PC | Windows PC | `http://127.0.0.1:3000` |
| Register #1 | Windows PC | `http://SERVER-IP:3000` |
| Register #2 | iPad PWA | `http://SERVER-IP:3000` |

**Critical Rule:** Register #1 is the physical cash drawer lane. All other lanes attach to Register #1's till group.

## Phase 2: Server Install (5-12 minutes)

### Install Backoffice/Server PC

**Step 1: Run Deployment Manager (2 minutes)**
- [ ] On Server PC, open unzipped deployment package
- [ ] Run `Start-RiversideDeployment.cmd`
- [ ] Choose "Backoffice / Server"
- [ ] Click **Check** and resolve any failed checks
- [ ] Click **Install**

**What the installer does:**
- Installs/uses PostgreSQL
- Creates Riverside database and user
- Copies server files and web/PWA assets
- Applies database migrations
- Downloads ROSIE AI model (~5.4 GB, may take extra time)
- Sets up firewall rule for port 3000
- Creates auto-start task
- Starts Riverside Server

**Step 2: Verify Server Health (3 minutes)**
- [ ] Open http://127.0.0.1:3000 on Server PC
- [ ] Staff sign-in screen loads
- [ ] Settings → Updates shows `0.85.5`
- [ ] Settings → General → About shows correct API host
- [ ] Settings → Data & Backups opens
- [ ] Backup path is writable

**Step 3: Test Network Access (2 minutes)**
- [ ] From another device, open `http://SERVER-IP:3000`
- [ ] Staff sign-in screen loads
- [ ] No version mismatch warning

**Server Repair Tools (if needed):**
| Tool | When to Use |
|------|-------------|
| Deployment Manager → Repair | Server files, firewall, task need refresh |
| Deployment Manager → PostgreSQL Status | Check/restart PG service |
| `Repair-RiversideCredentialsKey.cmd` | Settings says credentials key must be set |
| `Set-CounterpointBridgeToken.cmd` | Counterpoint bridge fails with 401 |

## Phase 3: Critical Integrations (12-18 minutes)

### Credential Key Setup (1 minute)

**Required before saving any integration credentials:**
- [ ] Sign in as administrator
- [ ] Navigate to Settings → any integration (e.g., Helcim)
- [ ] Try to save configuration
- [ ] If warning says `RIVERSIDE_CREDENTIALS_KEY must be set`:
  - [ ] Run `Repair-RiversideCredentialsKey.cmd` on Server PC (as admin)
  - [ ] Re-login and confirm integrations can be saved

### QuickBooks Online (3 minutes)

**For sales invoice and payment sync:**
- [ ] Settings → Integrations → QuickBooks
- [ ] Enter Client ID and Client Secret
- [ ] Click "Connect to QuickBooks"
- [ ] Sign in with store QBO account, select Company file
- [ ] Authorize connection
- [ ] Configure account mappings:
  - [ ] Sales Income Account
  - [ ] Cash/Tender Deposit Bank Account
  - [ ] Sales Tax Liability Account
  - [ ] Accounts Receivable (AR) Account
  - [ ] Inventory Asset Account
  - [ ] COGS Account
- [ ] Verify QBO Sync Worker status is `Idle` or `Syncing`

### Helcim Card Payments (3 minutes)

**For physical card terminal integration:**
- [ ] Settings → Integrations → Helcim
- [ ] Enter Merchant ID and API Access Token
- [ ] Set Environment to **Production** (unless testing)
- [ ] Configure terminal IP/device code for Register #1
- [ ] Trigger test ping, confirm "Terminal Connected"
- [ ] Copy webhook URL: `http://SERVER-IP:3000/api/webhooks/helcim`
- [ ] Configure in Helcim Developer Portal for events: `order.approved`, `transaction.captured`

### Shippo Shipping (2 minutes)

**For carrier rates and label printing:**

**For fast product/customer search:**

- [ ] Confirm URL: `http://127.0.0.1:7700`

## Common Issues and Fixes

### PWA will not load
1. Confirm device on correct network
2. Open `http://SERVER-IP:3000` in browser
3. Check DNS/server name
4. Check firewall on Server PC
5. Close/reopen Home Screen icon
6. If stale, remove icon, clear site data, re-add

### Register app won't reach server
1. Check Settings → General → About for API base
2. Confirm Register #1 NOT pointed at `127.0.0.1`
3. Open Server URL from Register #1
4. Run Repair from Deployment Manager

### PostgreSQL down/unreachable
1. Open Deployment Manager
2. Check PostgreSQL Status panel
3. If stopped, click **Start PG**
4. If connection failed, verify PG installed and running
5. Use **Restart PG** if service running but connections failing

### Receipt won't print
1. Check Printers & Scanners settings
2. For network mode, confirm Epson IP and port 9100
3. For installed mode, confirm Windows test page works
4. Restart desktop app after printer changes
5. For iPad, confirm server can reach network printer

### Update incomplete
1. Update Backoffice/Server PC first
2. Update Register #1 desktop app
3. On iPad, Check app files and reload
4. Do not continue until all surfaces show `0.85.5`

**Before live use:**
- [ ] Create fresh database backup
- [ ] Confirm backup file exists with non-trivial size
- [ ] Keep copy off server PC if practical
- [ ] Keep previous installer/build accessible
- [ ] Record backup timestamp and filename

3. Record reason for rollback
4. Restore database backup before running older binaries

## Post-Release Hotfixes (Applied to `main` after v0.85.5 tag)

The following bugs were identified during the v0.85.5 go-live and fixed in `main`. Pull a fresh copy of the deployment package from `main` or the next tag to include these fixes.

### Fix 1 — PostgreSQL Console Password Prompt Loop (`install-server.ps1`, `repair-bootstrap-admin.ps1`)

**Symptom:** During the server install (or repair), a PowerShell console window opened and repeatedly prompted for a PostgreSQL password. Entering the correct password had no effect.

**Root cause (three compounding bugs):**
1. `Invoke-NativeCommand` did not redirect stdin — psql could capture the open console and block waiting for input when `PGPASSWORD` failed.
2. All auth-probe `psql` calls were missing the `-w` (`--no-password`) flag; without it psql falls back to interactive prompting instead of failing immediately.
3. `$env:PGPASSWORD = $null` does not remove the variable in PowerShell (it sets it to an empty string); replaced with `Remove-Item Env:\PGPASSWORD`.

**Fix:** `RedirectStandardInput = $true` + immediate `Close()` in `Invoke-NativeCommand`; `-w` added to all psql probe calls; `PGPASSWORD` cleanup changed to `Remove-Item Env:\PGPASSWORD`.

---

### Fix 2 — ROSIE Binaries Missing; Empty LLM Model Path (`Install-RosieAiStack.ps1`)

**Symptom:** During server install, the output showed:
```
WARNING: Required ROSIE binary 'sherpa-onnx-offline.exe' is missing from the package
WARNING: ROSIE: LLM model is missing at ''. Skipping Riverside OS LLM Host scheduled task.
```
ROSIE showed as unavailable and `RIVERSIDE_LLAMA_MODEL_PATH` was empty in the server `.env`.

**Root cause:** The deployment ZIP never contains the sherpa-onnx binaries (large compiled executables are not in git). The old script threw a fatal exception when they were absent, causing `Install-RosieStack` to return `null` → empty model path → blank env var.

**Fix:** `Install-RosieAiStack.ps1` now auto-downloads all required components when not bundled:
- **sherpa-onnx v1.13.2** (Windows x64) from GitHub Releases → extracts `sherpa-onnx-offline.exe`, `sherpa-onnx-offline-tts.exe`, + DLLs
- **SenseVoice Small** STT model from HuggingFace
- **Kokoro-82M** TTS model from HuggingFace
- Gemma GGUF download failure is now a **warning** (not a crash) so STT/TTS work even if the model isn't downloaded yet.
