# Riverside OS v0.80.7 — Production Hardening: Phase 4 Resiliency Engineering

Riverside OS v0.80.7 is a major production hardening release focused on guaranteeing engineering-grade resilience under real-world retail pressure. It addresses high-risk transaction windows, network drops, offline queue conflicts, accounting integration decoupling, and webhook schema safety.

---

## What Changed

### 1. Close the "Transaction Window" Vulnerability (State Machine Hardening)
* **Processing Order Status**: Introduced a new `Processing` state to the `DbOrderStatus` enum to represent a checkout transaction in progress.
* **Client Idempotency Key**: Embedded a unique `checkoutClientId` generation in `useCartCheckout.ts` to prevent duplicate credit card sweeps during retries or network drops.
* **Atomic Pre-Payment Staging**: The POS registers checkout metadata and transitions the order state to `Processing` *before* initiating the external payment terminal transaction.
* **Webhook Auto-Recovery**: The terminal payment webhook (`handle_helcim_card_transaction` in `server/src/api/webhooks.rs`) matches approved payments against `Processing` checkouts using the `checkoutClientId`, automatically logs the payment, completes allocation, updates status to `Open` or `Fulfilled`, and creates the QBO outbox task.

### 2. Tightened Offline Queue Reconciler (Conflict Resolution)
* **Non-Blocking Sync**: Modified checkout inventory logic to allow physical stock quantities to drop below zero during offline replay rather than throwing database constraints that halt the sync engine.
* **Auto-Serialization & Alerting**: System logs over-allocated checkouts, inserts records into `negative_stock_alerts`, and broadcasts high-priority warnings to administrative staff via the notification system (`settings.admin`).

### 3. Bulletproof QBO Ledger Sync Bridge (Transactional Outbox Pattern)
* **Asynchronous Outbox**: Decoupled direct QBO API calls from the checkout transaction. Checkout runs write a ledger snapshot into a local `qbo_sync_outbox` table in the same DB transaction.
* **Background Outbox Processing**: The QBO background sync job (`server/src/jobs/qbo_sync.rs`) polls the outbox, formats ledger records into journal entries, handles expired OAuth sessions or API rate limits, and uses exponential backoff retries without interrupting checkout lanes.

### 4. Strict Validation Isolation Layers for Webhooks
* **Payload Validation**: Added strict structural validation layers using `serde` parsing for Podium, Helcim, and Shippo webhooks in `server/src/api/webhooks.rs`.
* **Graceful Failure**: Intercepts structural webhook payload schema shifts cleanly, logging structural errors without propagating panic behaviors into the core database loops.

## Deployment Notes

* **Workspace Version Parity**: All build manifests (`package.json`, Cargo workspaces, Tauri configurations, workflows, and deployment templates) are locked at version `0.80.7`.
* **Database Updates**: Database migrations are automatically checked and applied on server launch. Run script `./scripts/apply-migrations-psql.sh` or double-click `Apply-RiversideMigrations.cmd` on Windows.
