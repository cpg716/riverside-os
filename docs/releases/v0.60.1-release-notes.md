# Riverside OS v0.60.1 — Deployment Hotfix + ROSIE Availability Fix

Riverside OS v0.60.1 is a targeted hotfix for the production store rollout. It keeps the v0.60.0 release scope intact and delivers station setup reliability, PWA receipt-printer verification, Backoffice / Server host selection, updater readiness — and a fix that restores ROSIE AI assistant availability on the production server.

## What changed

### Station & printer setup
- Added known-host choices to **API Host Settings** on the sign-in screen, including a direct Backoffice / Server localhost option while still allowing a manually typed IP or DNS name.
- Added server-side printer readiness checks for PWA/browser receipt printing so the browser can verify the Riverside server can reach the receipt printer IP and port.
- Routed network Zebra tag readiness checks through the same direct path when the tag station is configured by IP.
- Kept reports printing on the browser/Windows print path.
- Removed the browser print fallback from PWA receipt/tag dispatch so hardware failures surface clearly instead of opening a misleading browser print window.
- Hardened installed-printer checks so a missing Windows printer selection shows a clear station setup error instead of a raw Tauri argument error.
- Updated Helcim, Shippo, and Counterpoint missing-credential messages to point operators to Backoffice Settings instead of `.env`-first setup.
- Optimized the normal Customer Hub browse path by paging first and computing expensive shipment/wedding/activity signals only for the visible page.
- Bumped release metadata to `0.60.1` across app, server, Tauri, and deployment package defaults.

### ROSIE AI assistant — production availability fix
- **Root cause**: The server's `RIVERSIDE_LLAMA_UPSTREAM` environment variable was never set automatically, so satellite clients (iPad, additional workstations) and server-side insight/search-intent calls had no path to the local LLM.
- **Server startup fix**: The Axum server now calls `ensure_rosie_upstream_from_local_llama()` at launch. If `RIVERSIDE_LLAMA_UPSTREAM` is not already set, it derives it from `RIVERSIDE_LLAMA_HOST:RIVERSIDE_LLAMA_PORT` (default `127.0.0.1:8080`) — the same address the Tauri sidecar listens on.
- **Model path fix**: Both the server and Tauri sidecar default paths now correctly reference the **Gemma 4 E2B** model and the Windows `%LOCALAPPDATA%` directory, matching `MODEL_PIN.json`. The previous default pointed to a non-existent E4B variant.
- **Installer fix**: `install-server.ps1` now includes a full ROSIE setup step that downloads the pinned Gemma GGUF (SHA256-verified from `MODEL_PIN.json`), installs `sherpa-onnx` via `uv`, and fetches the SenseVoice STT and Kokoro TTS models from k2-fsa releases. The resolved model path is written to the server `.env`. Use `-SkipRosieSetup` for air-gapped installs.
- **Bug Reports panel**: The diagnostic download action is now labelled **Download AI diagnostic** (was "Download for Codex") to be tool-agnostic.

## Deployment notes

- Backoffice / Server should use `http://127.0.0.1:3000` in the Windows desktop app.
- Register #1 and PWA devices should use the Backoffice / Server LAN or Tailscale URL, for example `http://10.64.70.196:3000`.
- PWA receipt printing remains server-mediated direct TCP to the configured Epson receipt printer IP and port.
- PWA tag printing uses the same server-mediated direct path when configured for a network Zebra printer.
- Reports continue to print through browser/Windows print.
- Integration credentials should be entered in Backoffice Settings. The Windows repair tools are only for restoring the server credential encryption key or synchronizing the Counterpoint bridge token when the installed server environment is broken.
- Windows updater assets must be produced by the signed Tauri updater workflow so `latest.json`, the MSI, and the `.sig` all point to `0.60.1`.
- **ROSIE on existing servers**: Run `Install-RosieAiStack.cmd` from the hotfix package on the Server PC. It downloads all required AI models, installs the Python voice runtime, patches the server `.env`, and restarts the server task. No full reinstall required.
