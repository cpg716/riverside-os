# Riverside OS v0.70.7 — Production Audit Probes, Negative Stock Warnings, Staff Avatar Photos, and Insights Reporting

Riverside OS v0.70.7 introduces integrated production audit probes in the ROS Dev Center, checkout-time negative stock alerts, a dedicated negative stock insights report, staff avatar photo upload with automatic face-aware processing, and corrected build documentation.

## What changed

### 1. Staff Avatar Photo Upload
* **Real Photo Upload**: Staff can upload actual employee photos in **Settings → Profile** (Back Office) or **Staff Profile** (POS).
  * Upload via drag-and-drop or file picker; supported formats: JPEG, PNG, WebP.
  * Max upload size: 10 MB.
  * **Face-aware processing**: Server-side face detection using skin-tone heuristics calculates a dynamic crop box including head and shoulders, enforces a 1:1 square aspect ratio, resizes to 512x512, and encodes as JPEG.
  * Photo URLs take precedence over avatar keys everywhere avatars are displayed: Top Bar, Register Overlay, Staff Roster, Staff Tasks, Notifications, Operational Dashboard, and Staff Search.
  * Photos are stored in `uploads/avatars/{staff_id}.jpg` and served as static files.
  * Staff can delete their photo to revert to the previously selected avatar icon.
* **API Endpoints**:
  * `POST /api/staff/{id}/avatar-photo` — upload and process a photo.
  * `DELETE /api/staff/{id}/avatar-photo` — remove the photo and revert to avatar key.
* **Auth Integration**: `avatar_photo_url` is returned in sign-in responses, effective-permissions refresh, and POS session payloads so the photo is available immediately after authentication.

### 2. Production Audit Probes in DevOps Center
* **13 Read-Only Probes**: Runs directly from Settings → ROS Dev Center → Production Audit Probes.
  * Duplicate checkout client IDs (critical)
  * Orphan payment allocations — missing payment tx or target tx (critical)
  * Over-allocated payment transactions (warning)
  * Register sessions reconciling >2 hours (warning)
  * Parked sales on closed register sessions (warning)
  * Negative available stock by variant (warning)
  * Order lines with premature stock decrement (warning)
  * Tax-exempt transactions missing reason (warning)
  * Finalized commission without fulfillment (warning)
  * Unbalanced QBO staging rows (warning)
  * QBO staging missing business_timezone (warning)
  * Stale backup health >30 hours (critical)
* **Historical Storage**: `ops_audit_probe_run` and `ops_audit_probe_result` tables store every execution with per-probe violation counts and detail rows.
* **Alert Integration**: Non-zero violations create/refresh a Dev Center alert (`audit_probe_failure`); clean runs auto-resolve it.
* **UI**: Run history (last 50), click-to-expand probe details, severity-colored rows, expandable JSON detail viewer.

### 3. Checkout Negative Stock Warning (Non-Blocking)
* `UPDATE product_variants ... RETURNING stock_on_hand, sku` during takeaway stock decrement.
* If `stock_on_hand < 0` after sale, warning collected and returned in `CheckoutResponse.warnings`.
* Client displays info toast after checkout success — sale always completes.

### 4. Negative Stock Insights Report
* **New Endpoint**: `GET /api/insights/negative-stock` returns up to 500 SKUs with negative stock.
* **Fields**: SKU, product name, category, vendor, stock_on_hand, reserved, available, unit_cost, extended_value (negative = overstated asset), reorder_point, last_sold_at, last_received_at.
* Added to ROSIE reporting specs as `"negative_stock"`.

### 5. Documentation Accuracy
* `DEPLOYMENT_MANAGER.md`: Corrected Windows build time claim from "8-10 minutes" to "20-30 minutes".

## Deployment notes

* All versions synchronized at `0.70.7` across root, client, server, Tauri, ros-dev, and Deployment Manager manifests.
* Windows deployment builds must use `build-deployment-package.ps1` at tag `v0.70.7`.
* macOS Deployment Manager DMG available on the GitHub release.
* Ensure target servers have applied migrations `040_ops_audit_probes.sql` and `041_staff_avatar_photo.sql`.
* Set `RIVERSIDE_GITHUB_TOKEN` (server-side, `repo` + `workflow` scopes) for in-app GitHub DevOps features.
* Ensure the `uploads/avatars` directory exists and is writable by the server process for photo storage.
