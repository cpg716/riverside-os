# Riverside OS v0.70.5 — Three-Role Deployment Architecture, macOS Deployment Manager, and DevOps Center

Riverside OS v0.70.5 introduces a unified three-role deployment architecture, a native macOS Deployment Manager companion, comprehensive removal scripts, and the DevOps Center for in-app and remote server management.

## What changed

### 1. Three-Role Deployment Architecture
* **Unified Installation Roles**: The Deployment Manager now presents three clear, mutually exclusive roles per workstation:
  * **Main Hub** — PostgreSQL, Riverside Server API, web bundle, migrations, firewall rule, startup task, and desktop app. One per store.
  * **Standalone App — Back Office** — Desktop app only, pointed at the Main Hub. For non-server back-office workstations.
  * **Standalone App — Register #1** — Desktop app only, pointed at the Main Hub, with receipt-printer and cash-drawer settings. Primary cashier station.
* **StationMode Parameter**: `install-register.ps1` now supports a `-StationMode` parameter (`Register` or `BackOffice`) so a single script handles both standalone workstation types.

### 2. Removal Scripts
* **`remove-main-hub.ps1`**: Fully removes a Main Hub installation — stops scheduled tasks (`Riverside OS Server`, `Riverside OS LLM Host`), terminates server/LLM processes, uninstalls the desktop app, removes firewall rules, drops the PostgreSQL `riverside_os` database (unless `--KeepDatabase`), and cleans the install root (unless `--KeepInstallRoot`).
* **`remove-standalone-app.ps1`**: Removes a desktop-app-only installation from Back Office or Register workstations — stops client processes, uninstalls via MSI/registry, and removes station configs. Does not touch PostgreSQL or server tasks.
* Both scripts require Administrator elevation and prompt for `REMOVE` confirmation (bypass with `--Force`).

### 3. macOS Deployment Manager Release
* **`.github/workflows/macos-deployment-manager-release.yml`**: New automated workflow that builds a universal Apple Silicon / Intel DMG on every `v*` tag push.
* **Published as Release Asset**: The DMG appears on the GitHub release alongside Windows artifacts.
* **Purpose**: macOS-based remote server management and DevOps monitoring without requiring a Windows VM.

### 4. CI/CD Automation Improvements
* **Auto-Trigger on Tag Push**: `windows-deployment-package.yml` now triggers automatically on `v*` tag push (in addition to manual dispatch).
* **Updater Asset Publishing**: The Windows workflow now builds and publishes the signed updater bundle (`latest.json`, `.sig`, updater artifact) directly to the GitHub release.
* **Shared Cache Key**: Added `shared-key: "windows-release"` to `swatinem/rust-cache` for consistent cache hits across Windows builds.
* **PowerShell Validation**: Added parse-validation step for all `deployment/windows/*.ps1` scripts using the .NET Language Parser.

### 5. DevOps Center
* **In-App ROS Dev Center** (`Settings → ROS Dev Center`): Operational status, runtime diagnostics, E2E health, GitHub workflow runs, releases, and one-click release builds for staff with `ops.dev_center.view` / `ops.dev_center.actions` permissions.
* **Standalone ROS Dev Center** (`ros-dev/`): macOS Tauri app with auto-discovery, server profiles, Tailscale detection, keychain-secure PIN storage, real-time dashboard, and ROSIE AI analysis.
* **Server-side API Proxy**: `GET /api/ops/github/workflows`, `GET /api/ops/github/releases`, `POST /api/ops/github/dispatch` protected by `ops_shield_middleware` (loopback, RFC 1918, and Tailscale only).

### 6. Deployment Manager Hardening
* **Zero-Config Credentials**: Auto-generates JWT secrets, app passwords, and PostgreSQL admin passwords when blank or placeholder.
* **Factory Reset**: Single-click "Start Fresh" drops and recreates the database, applies migrations, and seeds core data — with WinForms MessageBox suppression for headless Tauri execution.
* **Repair Commands**: `repair-server-credentials-key.ps1` and `repair-bootstrap-admin.ps1` for credential recovery and lockout reset.

## Deployment notes

* All versions are synchronized at `0.70.5` across root, client, server, Tauri, ros-dev, and Deployment Manager manifests.
* Windows deployment builds must be generated using `build-deployment-package.ps1` at the tag `v0.70.5`.
* macOS Deployment Manager DMG is available on the GitHub release for remote management.
* Ensure target servers have applied the active migration schema baseline using `apply-migrations-psql.sh` or the Deployment Manager graphical interface.
* Set `RIVERSIDE_GITHUB_TOKEN` (server-side only, `repo` + `workflow` scopes) to enable in-app GitHub DevOps features.
