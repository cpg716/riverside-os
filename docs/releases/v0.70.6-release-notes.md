# Riverside OS v0.70.6 — Negative Stock Warnings, Insights Reporting, and Documentation Accuracy

Riverside OS v0.70.6 builds on the v0.70.5 deployment architecture with checkout-time negative stock alerts, a dedicated negative stock insights report, and corrected build time documentation.

## What changed

### 1. Checkout Negative Stock Warning (Non-Blocking)
* **RETURNING Clause**: The takeaway stock decrement `UPDATE` now uses `RETURNING stock_on_hand, sku` to read the post-sale inventory level.
* **Warning Collection**: If `stock_on_hand < 0` after decrement, a warning like `"ABC-123 stock went negative after sale (now -5)"` is collected.
* **Client Toast**: `useCartCheckout.ts` reads the `warnings` array from the checkout response and displays each warning as an info toast after the success message.
* **Sale Completes**: The transaction always finalizes — the warning is informational for the cashier, not a blocker.

### 2. Negative Stock Insights Report
* **New Endpoint**: `GET /api/insights/negative-stock` returns up to 500 SKUs with negative `stock_on_hand`.
* **Row Details**: `sku`, `product_name`, `category_name`, `vendor_name`, `stock_on_hand`, `reserved_stock`, `available_stock`, `unit_cost`, `extended_value` (`cost * stock_on_hand`, negative = overstated asset), `reorder_point`, `last_sold_at`, `last_received_at`.
* **ROSIE Integration**: Added to ROSIE reporting specs as `"negative_stock"` — callable via `POST /api/ai/rosie-reporting-run`.
* **Complements Exception Risk**: The existing `/api/insights/exception-risk` already counts negative stock SKUs; this new endpoint provides the SKU-level detail needed for reconciliation.

### 3. Documentation Accuracy
* **DEPLOYMENT_MANAGER.md**: Corrected misleading claim that Windows deployment package builds take "8-10 minutes". Actual runtime is **20-30 minutes** for three sequential Rust release compiles on a standard GitHub runner. macOS builds at ~15 minutes remain accurate.

### 4. Version Synchronization
* All manifests bumped from `0.70.5` to `0.70.6`: root `package.json`, `client/package.json`, `server/Cargo.toml`, `client/src-tauri/Cargo.toml`, `client/src-tauri/tauri.conf.json`, `deployment/manager-app/package.json`, `deployment/manager-app/src-tauri/Cargo.toml`, `deployment/manager-app/src-tauri/tauri.conf.json`, `ros-dev/package.json`, `ros-dev/src-tauri/Cargo.toml`, `ros-dev/src-tauri/tauri.conf.json`, `deployment/windows/riverside-deployment.config.example.json`, `README.md`.

## Deployment notes

* All versions are synchronized at `0.70.6` across root, client, server, Tauri, ros-dev, and Deployment Manager manifests.
* Windows deployment builds must be generated using `build-deployment-package.ps1` at the tag `v0.70.6`.
* macOS Deployment Manager DMG is available on the GitHub release for remote management.
* Ensure target servers have applied the active migration schema baseline using `apply-migrations-psql.sh` or the Deployment Manager graphical interface.
* Set `RIVERSIDE_GITHUB_TOKEN` (server-side only, `repo` + `workflow` scopes) to enable in-app GitHub DevOps features.
