# Riverside OS v0.95.0 Release Notes

- Status: **Published — Latest**

## Latest rebuild

- **ROSIE universal search reliability** — stabilized the receipt-order navigation callback so search state updates do not restart the same debounced query or submit duplicate ROSIE search-intent requests.
- **Focused validation** — the complete phone/tablet responsive Playwright spec passed 10/10, including the previously failing ROSIE intent request contract; client lint, TypeScript, Rust formatting, and Help-impact review also passed.

## Why v0.95.0

Riverside OS changed substantially during the v0.90.0 development and repeated hardening cycle. Moving directly to v0.95.0 makes that distance visible while preserving v1.0.0 for the formal production-maturity milestone. This is a new versioned release, not another same-version rebuild of v0.90.0.

## Major advances represented by v0.95.0

- **Financial and checkout integrity** — exact-cent tax and total parity, stronger tender and recovery boundaries, auditable deposit/payment allocation, QBO mapping and liability corrections, and safer post-checkout recovery.
- **Wedding party payments** — split wedding deposits are visible on the beneficiary customer, notify Register staff, can be applied during final payment, and retain correct liability, allocation, void, and fulfillment accounting.
- **Help Center and ROSIE** — expanded staff manuals and screenshots, searchable ROSIE Help coverage, and an on-demand printable RiversideOS User Manual PDF generated from the current effective manuals.
- **Register and printing reliability** — hardened native Tauri routing for receipts, Reports, and Riverside LP 2844 EPL tags, with bounded dispatch, retry visibility, and truthful success messaging.
- **Counterpoint transition tooling** — direct Bridge-to-ROS import proof, run-scoped provenance and exceptions, stronger catalog/open-document recovery, and clearer Main Hub operational status.
- **Provider and operational resilience** — hardened Helcim, QBO, IMAP, Shippo, connectivity, backup/update, offline recovery, and transactional outbox boundaries without placing remote provider waits inside customer-money locks.
- **Windows deployment and updates** — rollback-safe Main Hub installation, stronger migration/bootstrap recovery, signed standalone updaters, exact-build deployment selection, and clearer Deployment/Server Manager diagnostics.
- **Release engineering** — Node 24 and Rust 1.91 build tooling, immutable action pins, bounded Dependabot load, per-job Rust caches, concurrent Windows/macOS builds, aggregate Playwright enforcement, and exact-SHA release-candidate promotion.

## Release method

v0.95.0 must use the verified candidate path:

1. Commit the complete v0.95.0 source and documentation with all version manifests in parity.
2. Pass Lint Checks and blocking Playwright on that exact commit.
3. Run the Windows and macOS release workflows with `publish_release=false`.
4. Confirm both candidate runs completed successfully and retained all required signed artifacts.
5. Create `v0.95.0` on that exact candidate commit.
6. Run **Promote verified release candidates** with both run IDs.
7. Verify published updater manifests, build manifests, signatures, deployment ZIP, release title/body, and installed-machine update behavior.

The promotion workflow fails closed when the tag SHA, Windows run SHA, macOS run SHA, GitHub artifact digests, updater build manifests, or deployment package provenance do not agree.
