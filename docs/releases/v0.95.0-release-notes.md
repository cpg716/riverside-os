## Main Hub update recovery and pickup-only rebuild (2026-07-24)

- **Migration 159 checksum compatibility restored** — the canonical migration
  file is restored byte-for-byte to SHA-256
  `03bf999619095b95fa4e4e0f76cfb37042cb5d415b48c3ef1f21dd35f29a28f4`,

  lines for pickup loads only those selected items into the Register and does
  not automatically add the Transaction Record balance as a payment. Staff use

  cumulative recorded payments against merchandise already released plus the
  selected pickup value. When coverage is short, staff may collect payment or
  use explicit Manager Access to release the selected items. The approving
  manager, reason, selected value, recorded payments, and shortage are retained
  in the pickup audit.
- **Partial work remains open** — an approved one-item pickup marks only that
  selected line picked up. Unselected lines, their fulfillment work, and the
  remaining Transaction Record balance stay open.
- **Focused release evidence** — the isolated pickup/shipping certification
  contract passed both tests, including an unpaid selected-item Manager
  override that leaves another line and the balance open. TypeScript, Rust,
  financial-invariant, Help-impact, formatting, and targeted lint checks also
  passed. The complete exact-commit release Playwright and packaging workflows
  remain required before the rebuilt assets are certified.

## Register Helcim sale-isolation rebuild (2026-07-23)

- **Prior sales cannot reserve a new checkout** — Helcim attempt ownership now requires the exact Register session and checkout identity. Terminal availability counts only pending attempts from an open Register session, and a pending attempt retained under a closed or missing session is expired for provider review before a new terminal request is inserted.
- **Current-sale payment safety remains intact** — an exact pending, approved, or captured attempt for the active checkout still requires a definitive provider outcome or attachment. The isolation change does not discard provider evidence, infer a charge result, or allow an unresolved payment from the current sale to be replaced silently.
- **Register warnings are scoped to the active Register** — the live badge reads only the open POS session and filters retained local queue rows by that same session. Historical/global recovery remains available in Payments and close history without entering a new checkout or disabling its payment controls.
- **Customer and pickup state cannot cross sales** — selecting a different customer clears the prior customer's pickup Transaction Record, selections, paid amount, ready alterations, and confirmation before the new checkout can open.
- **Order-payment retry is idempotent** — when an approved Helcim order payment was already normalized, allocated once to the exact customer Transaction Record, and linked to its zero-dollar ledger shell, a repeated Register completion returns that committed result instead of creating a false Checkout Recovery item or failing because the target is already closed.
- **Fulfilled balances remain payable** — fulfillment status alone no longer rejects an order payment when the exact customer Transaction Record still has a positive balance. Zero-balance, canceled, wrong-customer, and overpayment targets remain blocked, and an approved retained attempt can be replayed through the original audited checkout instead of charging the card again.
- **Exact recovery preserves multi-target ledgers** — Match Existing Paid Transaction validates every immutable allocation from the original checkout, including a tender divided between the current sale and existing order payments. A proven stale pickup cross-link can be canceled with Manager Access and retained audit evidence without moving money or changing fulfillment.
- **Resolved warnings clear across Register sessions** — an older Register build can reread an exact resolved checkout recovery from the same station after the Main Hub verifies its immutable checkout and committed Transaction Record evidence. The rebuilt client also performs an exact Staff-authorized lookup when current-session mirroring is correctly rejected, so resolved prior-session warnings leave the local badge without deleting their server audit record.
- **Audited production cleanup** — historical attempts were retained with recovery evidence while contradictory terminal reservations were finalized only after exact no-charge, no-provider-match, or signed terminal-cancel proof. Verified already-recorded order payments were left unchanged and their stale recovery rows were completed without creating another payment or allocation.
- **Migration integrity** — migrations 150–156 preserve the incident evidence, make production-specific repairs no-op on databases without those exact rows, require both provider identifiers to be absent for generic finalization, and isolate closed-session pending terminal reservations from new sales.
- **Validation boundary** — targeted Rust, TypeScript, lint, migration-layout, Help-impact, and release gates are required for this rebuild. Local Playwright E2E is intentionally skipped at operator direction; exact-commit GitHub Playwright remains the publication gate.

## Register, search, reporting, and runtime integrity rebuild (2026-07-23)

- **Sale completion with durable recovery** — Register checkout now locks and revalidates the active Register session, binds every attempt to exact request and tender fingerprints, and preserves the completed financial Transaction Record when a later audit-sync, pickup, or receipt-print follow-up cannot finish. The issue remains visible as a server-backed recovery task instead of deleting the sale or silently reporting success. A successful authoritative Main Hub reread clears a stale local online-unconfirmed blocker without treating an unavailable server as an empty result, and each completed/cleared cart drops its sale-level operator credential so the next sale must identify its actual cashier.
- **Atomic, job-scoped auditability** — offline checkout, unconfirmed checkout, paid pickup follow-up, receipt printing, exchange settlement, Manager replay, and post-close recovery use exact job identities, immutable checkout evidence, idempotent audit keys, and atomic finalization. Stale `pending` updates cannot downgrade a `blocked` recovery, concurrent retries cannot duplicate the sale, and an authorized ordinary Z-close remains available without hiding or resolving unresolved workstation or Main Hub recovery evidence.
- **Checkout drawer and Card Not Present sale isolation** — completing or clearing a sale creates a new checkout identity and remounts the checkout drawer with clean keypad, tender, refund, gift-card, and hosted-card state. A Card Not Present handoff is accepted only when its request nonce, provider attempt, checkout, customer, and amount all match the active sale; delayed messages and approvals from a prior sale are rejected rather than copied into the next customer's ledger.

## Refund, exchange, and payment integrity (2026-07-17)

This replacement closes the refund and exchange gaps found during live Register use:

- **Provider-correct Helcim refunds** — linked card refunds verify the current Helcim transaction and batch state, reverse only full open-batch purchases, refund closed-batch purchases, and fail closed when provider state cannot be proven.
- **No premature financial or inventory writes** — return, restock, exchange, and refund ledger changes stay uncommitted until the provider approves the card action. Approved-provider retries reconcile the ROS ledger without charging or refunding Helcim twice.
- **Real tender-specific refunds** — cash, check, gift card, store credit, Staff Account, RMS Charge, RMS 90, and legacy/manual Helcim card refunds follow their actual ledger and authorization paths. Unsupported or synthetic tenders are rejected before a payment row is written.
- **Exchange completion durability** — an exchange settles its returned and replacement lines, inventory movements, tax, and reporting first. If a card remainder cannot complete, the financial remainder stays visible in the refund queue for canonical retry instead of rolling back the real exchange.
- **Receipt and reporting truth** — every returned item remains a negative line and every replacement remains a positive line. Effective quantities, original paid price and tax, QBO liability clearing, COGS reversal, inventory restoration, and staff-facing receipts stay aligned.
- **Payment-only completion clarity** — payments against existing Transaction Records use the distinct **Payment recorded** completion state and retain the target transaction, amount, and remaining balance evidence.
- **Small-screen command access** — the secondary ROSIE header action now yields below 480 px so universal search remains clickable on phone-sized displays.

Riverside OS changed substantially during the v0.90.0 development and repeated hardening cycle. Moving directly to v0.95.0 makes that distance visible while preserving v1.0.0 for the formal production-maturity milestone. This is a new versioned release, not another same-version rebuild of v0.90.0.

## Release method

Any same-version replacement of this candidate must use the guarded retag path:

1. Commit the complete v0.95.0 source and documentation with version manifests in parity.
2. Pass the unified pre-retag gate. Local serialized Playwright may be skipped only by explicit operator direction; exact-commit GitHub Playwright remains blocking before publication.
3. Move `v0.95.0` to the exact clean commit with `npm run release:retag` and refresh the GitHub Release body.
4. Pass exact-commit Lint Checks and all four blocking Playwright shards.
5. Publish signed Windows and macOS assets only after the same-commit Playwright gate passes.
6. Verify updater manifests, build manifests, signatures, release target, and the short-SHA Windows deployment ZIP independently after publication.

The release workflows fail closed when the tag target, workflow SHA, required artifacts, updater manifests, or deployment-package provenance do not agree.
