## Register, search, reporting, and runtime integrity rebuild (2026-07-22)

- **Sale completion with durable recovery** — Register checkout now locks and revalidates the active Register session, binds every attempt to exact request and tender fingerprints, and preserves the completed financial Transaction Record when a later audit-sync, pickup, or receipt-print follow-up cannot finish. The issue remains visible as a server-backed recovery task instead of deleting the sale or silently reporting success. A successful authoritative Main Hub reread clears a stale local online-unconfirmed blocker without treating an unavailable server as an empty result, and each completed/cleared cart drops its sale-level operator credential so the next sale must identify its actual cashier.
- **Atomic, job-scoped auditability** — offline checkout, unconfirmed checkout, paid pickup follow-up, receipt printing, exchange settlement, Manager replay, and post-close recovery use exact job identities, immutable checkout evidence, idempotent audit keys, and atomic finalization. Stale `pending` updates cannot downgrade a `blocked` recovery, concurrent retries cannot duplicate the sale, and Z-close cannot hide unresolved workstation or Main Hub recovery evidence.
- **Checkout drawer and Card Not Present sale isolation** — completing or clearing a sale creates a new checkout identity and remounts the checkout drawer with clean keypad, tender, refund, gift-card, and hosted-card state. A Card Not Present handoff is accepted only when its request nonce, provider attempt, checkout, customer, and amount all match the active sale; delayed messages and approvals from a prior sale are rejected rather than copied into the next customer's ledger.
- **Truthful payment recovery and review** — **Clear Sale** releases sale-local Register state but never deletes the processor attempt from Payments Health. An empty Main Hub open-recovery list cannot erase local unconfirmed evidence, and local evidence clears only when Riverside returns the exact checkout, recovery job, and completed Transaction Record. When the paid Transaction Record already exists, **Match Existing Paid Transaction** verifies immutable checkout, customer, Register session, amount, currency, final provider status, payment allocation, and fingerprints before closing only the recovery record; it creates no sale, charge, refund, payment, or payment movement. A Payments Health review note remains informational and cannot hide an approval that is still unlinked.
- **Tender and exchange integrity** — exchange and refund paths retain exact original merchandise/tax components, staff attribution, provider identity, and replacement-settlement intent. A manually recorded external-card refund now uses a short-lived, server-issued Manager approval bound to the Transaction Record, Register session, exact amount, external reference, reason, and card last four; the Access PIN is never forwarded into the refund write. Invalid totals, mismatched identities, unsupported tenders, and missing authorization remain explicit safety stops; recoverable post-payment work has a documented Manager completion path.
- **PIN and secret containment** — Access PIN, Manager PIN, Staff PIN, normalized aliases, and PIN keys nested inside JSON-encoded strings are removed from payment, allocation, outbox, recovery, and Staff Access audit payloads. Historical checkout fingerprints derived from exposed PIN payloads are retired rather than treated as replay authority.
- **Truthful Register close and reporting** — Register #1 owns the till-close group, linked lanes acknowledge their queue state, and forced close requires Manager Access while preserving post-close evidence. Daily Sales, Z-Reports, print, and CSV output use canonical cents-based ledger values, stable full-period totals, exact paging/count checks, and explicit Booked versus Fulfilled bases. Each interactive page is calculated inside a read-only repeatable-read transaction, while complete View/Print/CSV and Z-close output are assembled under one bounded database snapshot so concurrent sales cannot create a mixed-time report.
- **Counterpoint booking and financial evidence** — ticket imports now copy the retained Counterpoint booking time into every imported line and initial booking event, reject missing/invalid source dates instead of substituting the import time, store header/line/tender comparison evidence, and open review exceptions when those sources disagree. Reruns retain superseded booking evidence without manufacturing current-day line-deletion activity. Migration 146 adds a read-only financial-integrity report and dry-run booking-date repair manifest; the guarded Manager repair records before/after evidence and cannot change transaction headers, payments, allocations, or tender amounts. Older imported Transaction Records receive a truthful synthesized import-evidence row in the audit feed without rewriting history.
- **Fast, authoritative search** — universal and workspace searches cancel superseded requests, reject punctuation-only input, treat SQL wildcards literally, enforce bounded deadlines, and report partial-source failures. Fifteen PostgreSQL trigram indexes accelerate the guarded fallback path; Meilisearch is authoritative only after a recent complete rebuild with healthy index/count parity, otherwise search remains functional through PostgreSQL.
- **Runtime and backup truth** — readiness now distinguishes required database/job-worker failures from optional Redis degradation, metrics omit unsupported measurements instead of publishing false zeros, and telemetry collection no longer holds registry locks across network/database waits. Docker backup fallback now targets the exact database parsed from `DATABASE_URL`, preventing a version-mismatch fallback from dumping the wrong database. Cloud backup success requires a bounded read-back of the finalized object whose byte length and SHA-256 match the uploaded stream; an unverifiable object is removed and reported as failed.
- **Cross-restart updater truth** — the desktop updater persists the target version/build before installation begins and reports an observed install timestamp only after a later app launch runs that exact version/build. Download failures and stale mismatched relaunches become durable failed observations; browser/PWA behavior remains unchanged.
- **Migration integrity** — migrations 143 and 144 add Register recovery/fingerprint evidence, exchange settlement identity, historical PIN cleanup, and deletion-safe transaction-line booking events. Migration 145 restores idempotent order, alteration, and appointment notification enqueueing after the active-row unique index became partial. Migration 146 adds non-destructive Counterpoint financial-integrity and booking-date repair evidence, migration 147 separates independently verified backup evidence from scheduler status, migration 148 adds normalized Receiving identifier indexes, and migration 149 binds readiness to the exact verified backup filename, byte length, and SHA-256 identity. The active migration baseline and embedded runner agree through version 149.
- **Candidate validation** — a fresh isolated PostgreSQL database applied all 148 active migrations sequentially through version 149 with complete checksums and a passing schema contract. Rust formatting, migration-layout and whitespace validation, Riverside server and Tauri compilation, the full seeded server library suite (586 passed, 12 intentionally ignored, 0 failed), 23 backup tests, 7 readiness tests, focused Counterpoint/Receiving/Helcim/report database regressions, client typecheck/lint/production build, Deployment Manager lint/build with zero npm advisories, 84 go-live gates, 115 financial gates, 38 report definitions, 15 search-index definitions with 22 source checks, and Help impact are green. Targeted Register/Search/Receiving browser contracts passed 29 tests across their earlier final runs, and the live local offline-recovery stack passed 8/8 before this follow-up. Local Playwright is intentionally skipped for the final checkout/Card Not Present follow-up at operator direction; the unified pre-retag non-E2E gates and exact-commit GitHub Playwright remain publication requirements, and historical results below do not certify this source candidate.
- **Operational boundary at source freeze** — these source fixes had not yet been retagged, published, installed on the Main Hub, or verified against live production data. A verified production backup, exact-commit Playwright/release gates, signed asset publication, updater/build-manifest verification, and physical Main Hub/terminal/printer checks remain required before this candidate can be called a production fix.

## Exact refund accounting and external card refunds (2026-07-20)

This replacement makes each return/refund a first-class financial event instead of reconstructing it from sale totals or display values:

- **Exact return event ledger** — every returned line records its exact merchandise, state-tax, local-tax, and total refund components under one refund event. Mismatched totals fail closed, and inventory restoration remains tied to the real returned quantities.
- **Manual external credit-card refund** — staff can record a card refund completed on a prior processor as a real negative card tender. Manager Access, the external processor reference, reason, and card last four are required; ROS does not fabricate a Helcim approval or call Helcim for this path.
- **Automatic Helcim refund boundary** — refund-to-original-card is offered only when the original ROS payment has a verified Helcim provider transaction ID. Imported or externally processed cards use the manual external-card workflow.
- **Receipt and completion accuracy** — refund receipts, reprints, and the completion screen show each returned item as a negative merchandise line, the exact refunded tax, the actual negative tender, and the real refund total rather than `$0.00` pickup semantics.
- **Daily, Z-Report, and QBO parity** — return-day activity, Register reconciliation, credit-card totals, QBO revenue/tax reversals, refund liability, and tender clearing consume the same refund event and payment ledger.
- **Focused validation** — Rust compilation, client typecheck, targeted POS lint, Help impact, exact refund contracts, receipt builders, QBO tests, and Register reporting tests passed. Local Playwright E2E was intentionally skipped for this rebuild at operator request; exact-commit GitHub Playwright remains a publication gate.

## Card reporting and Payments workspace follow-up (2026-07-20)

This replacement corrects card-entry reporting and expands the built-in Payments workspace:

## Refund, exchange, and payment integrity (2026-07-17)

This replacement closes the refund and exchange gaps found during live Register use:

- **Provider-correct Helcim refunds** — linked card refunds verify the current Helcim transaction and batch state, reverse only full open-batch purchases, refund closed-batch purchases, and fail closed when provider state cannot be proven.
- **No premature financial or inventory writes** — return, restock, exchange, and refund ledger changes stay uncommitted until the provider approves the card action. Approved-provider retries reconcile the ROS ledger without charging or refunding Helcim twice.
- **Real tender-specific refunds** — cash, check, gift card, store credit, Staff Account, RMS Charge, RMS 90, and legacy/manual Helcim card refunds follow their actual ledger and authorization paths. Unsupported or synthetic tenders are rejected before a payment row is written.
- **Exchange completion durability** — an exchange settles its returned and replacement lines, inventory movements, tax, and reporting first. If a card remainder cannot complete, the financial remainder stays visible in the refund queue for canonical retry instead of rolling back the real exchange.
- **Receipt and reporting truth** — every returned item remains a negative line and every replacement remains a positive line. Effective quantities, original paid price and tax, QBO liability clearing, COGS reversal, inventory restoration, and staff-facing receipts stay aligned.
- **Payment-only completion clarity** — payments against existing Transaction Records use the distinct **Payment recorded** completion state and retain the target transaction, amount, and remaining balance evidence.
- **Small-screen command access** — the secondary ROSIE header action now yields below 480 px so universal search remains clickable on phone-sized displays.

## Follow-up reliability hardening (2026-07-16)

This replacement build also includes the production-stability fixes identified from the Main Hub diagnostics:

- **Checkout recovery and cart durability** — successful provider-backed checkouts now create an operational recovery record when the client loses the final response, and queued cart persistence cannot restore a completed sale after the register clears it.
- **Payment-screen recovery** — pending Card Not Present and terminal attempts always expose cancellation/alternate-tender recovery, including a warning when provider cancellation could not be confirmed.
- **Helcim provider backoff** — HTTP 429 responses are not retried immediately; scheduled fee and settlement syncs stop after throttling and pause for 15 minutes so provider rate limits cannot cascade into repeated 502s.
- **Main Hub connection resilience** — authenticated rate limits are isolated by station/session rather than shared only by LAN IP, and Redis health probes fail fast because Redis is an optional acceleration layer.
- **History and read-path stability** — customer activity timestamp aggregates are explicitly typed as timestamps to prevent the production `max(uuid)` failure; migration 132 adds payment/history and wedding lookup indexes for the slow diagnostic queries.
- **Receipt and payment documentation** — the POS manual now documents the alternate-tender recovery path and accurately describes split wedding deposit receipt totals.
- **Transaction-truthful POS completion** — shipping and alteration charges, order payments, pickups, and mixed sales remain visible in the cart and checkout summary; receipts and Sale Complete now use the actual amount collected, paid/balance ledger, customer, and transaction context rather than the original fulfillment-order total. Payment-only receipt allocations remain distinct and identical tender rows are preserved for reconciliation.
- **Helcim sale-attachment guard** — checkout now refuses to record a sale while an approved Helcim attempt for the same retained checkout remains unattached, preventing an apparently completed sale from being saved without its card payment; staff can attach the exact tender or use Payments Health to recover or refund it.

## Why v0.95.0

Riverside OS changed substantially during the v0.90.0 development and repeated hardening cycle. Moving directly to v0.95.0 makes that distance visible while preserving v1.0.0 for the formal production-maturity milestone. This is a new versioned release, not another same-version rebuild of v0.90.0.

## Major advances represented by v0.95.0

- **Financial and checkout integrity** — exact-cent tax and total parity, stronger tender and recovery boundaries, auditable deposit/payment allocation, QBO mapping and liability corrections, and safer post-checkout recovery.
- **Wedding party payments** — split wedding deposits are visible on the beneficiary customer, notify Register staff, can be applied during final payment, and retain correct liability, allocation, void, and fulfillment accounting.
- **Help Center and ROSIE** — expanded staff manuals and screenshots, searchable ROSIE Help coverage, and an on-demand printable RiversideOS User Manual PDF generated from the current effective manuals.
- **Register and printing reliability** — hardened native Tauri routing for receipts, Reports, and Riverside LP 2844 EPL tags, with bounded dispatch, retry visibility, and truthful success messaging.
- **Counterpoint transition tooling** — direct Bridge-to-ROS import proof, run-scoped provenance and exceptions, stronger catalog/open-document recovery, and clearer Main Hub operational status.
- **Provider and operational resilience** — hardened Helcim, QBO, IMAP, Shippo, connectivity, backup/update, offline recovery, and transactional outbox boundaries without placing remote provider waits inside customer-money locks.
- **Windows deployment and updates** — rollback-safe Main Hub installation, stronger migration/bootstrap recovery, signed standalone updaters, exact-build deployment selection, and clearer Deployment/Server Manager diagnostics.

## Release method

Any same-version replacement of this candidate must use the guarded retag path:

1. Commit the complete v0.95.0 source and documentation with version manifests in parity.
2. Pass the unified pre-retag gate. Local serialized Playwright may be skipped only by explicit operator direction; exact-commit GitHub Playwright remains blocking before publication.
3. Move `v0.95.0` to the exact clean commit with `npm run release:retag` and refresh the GitHub Release body.
4. Pass exact-commit Lint Checks and all four blocking Playwright shards.
5. Publish signed Windows and macOS assets only after the same-commit Playwright gate passes.
6. Verify updater manifests, build manifests, signatures, release target, and the short-SHA Windows deployment ZIP independently after publication.

The release workflows fail closed when the tag target, workflow SHA, required artifacts, updater manifests, or deployment-package provenance do not agree.
