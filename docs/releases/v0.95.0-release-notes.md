## Latest rebuild

> **Windows deployment hold (2026-07-13):** Do not use the published `86724e73` Windows deployment package for another Main Hub update. Production-machine verification exposed an administrator-vs-app backup permission error followed by incomplete rollback credential recovery. The Main Hub was restored on v0.90.0 without database loss. A replacement v0.95.0 build must pass the hardened backup, rollback, database-readiness, exact-SHA, and signed-candidate gates documented below before this hold is removed.

- **Single-day Z-Report recovery** — Register close now resolves and closes the oldest unclosed store-local business date, never combines multiple days, and preserves each day's tenders, transactions, adjustments, inventory activity, QBO staging, deposit evidence, and Daily Financial Report separately. Migration 126 repairs the July 10 and July 11 production reports as distinct business days without inventing a historical drawer count.
- **Main Hub replacement-package safety** — all update paths now require a verified PostgreSQL administrator backup before stopping the existing server or replacing files, reject incomplete dumps, encode database credentials safely, and attempt prior-task restoration/restart independently when later rollback work fails.
- **Line-level tax controls** — Register sale lines preserve explicit Standard, Clothing, and No Tax selections through pricing, discounts, checkout, server validation, audit metadata, and persisted sale detail; shipping and alteration labor remain server-enforced non-taxable charges.
- **Legacy Counterpoint order repair** — Settings can now scan the historical Counterpoint data already stored in ROS, reconcile only exact order/ticket/payment matches into the original customer order, retain an audit snapshot, and leave ambiguous accounts unchanged for Manager review without running another import.
- **Production hardening across financial and data writes** — wedding group-pay allocations now lock and re-check beneficiary balances; shipping, refund, exchange, financial-date, status, and order-line mutations keep required audit history inside the same database transaction; strict production startup fails closed when employee pricing cannot be loaded.
- **Staff Access session integrity** — replaced retained raw-PIN authentication with hashed opaque sessions bound to the workstation and individual app/tab connection, including expiry, independent revocation, PIN/deactivation invalidation, active-session Station Fleet evidence, and a dedicated sign-in rate limit.
- **Staff Access expiry reliability** — long expiry intervals are scheduled in browser-safe chunks and rechecked, so clock skew cannot overflow the timer and immediately revoke a valid new session.
- **Connection performance and desktop/PWA safety** — warmed and bounded PostgreSQL pooling, added shared outbound HTTP keepalive/timeouts, throttled Staff/Register activity writes, protected the Tauri WebView with an explicit CSP, and kept PWA/LAN/Tailscale reconnect monitoring adaptive and non-blocking.
- **Register audit continuity** — Z-close reconciliation now uses scoped Staff/Register sessions instead of a retained PIN, close/handoff actions resolve the real authenticated staff actor, and post-sale commission-attribution corrections request a fresh Access PIN.
- **Backup, migration, and failure visibility** — migration 125 adds revocable Staff Access sessions; the backup/restore drill recognizes only approved test sandboxes and verifies the latest migration; Counterpoint export, staff-cap, and Register-search failures are now visible instead of appearing successful or empty.
- **Documentation and permanent gates** — refreshed Staff, POS, Remote Access, deployment, security, and production-hardening guidance; added focused session/reconciliation contracts; and expanded the permanent go-live gate to 80 checks.
- **ROSIE universal search reliability** — stabilized the receipt-order navigation callback so search state updates do not restart the same debounced query or submit duplicate ROSIE search-intent requests.
- **Focused local validation** — fresh and dirty-database migrations through 126 passed with no checksum drift; the unified pre-retag gate passed version parity, deployment/release guards, all 80 go-live blockers, Rust formatting and migration parser tests, client lint/typecheck, Help generation/impact, and whitespace checks. Exact-SHA GitHub Lint and aggregate Playwright remain required before replacement candidates are built.

## Why v0.95.0

Riverside OS changed substantially during the v0.90.0 development and repeated hardening cycle. Moving directly to v0.95.0 makes that distance visible while preserving v1.0.0 for the formal production-maturity milestone. This is a new versioned release, not another same-version rebuild of v0.90.0.

## Major advances represented by v0.95.0

- **Financial and checkout integrity** — exact-cent tax and total parity, stronger tender and recovery boundaries, auditable deposit/payment allocation, QBO mapping and liability corrections, and safer post-checkout recovery.
- **Wedding party payments** — split wedding deposits are visible on the beneficiary customer, notify Register staff, can be applied during final payment, and retain correct liability, allocation, void, and fulfillment accounting.
- **Help Center and ROSIE** — expanded staff manuals and screenshots, searchable ROSIE Help coverage, and an on-demand printable RiversideOS User Manual PDF generated from the current effective manuals.
- **Register and printing reliability** — hardened native Tauri routing for receipts, Reports, and Riverside LP 2844 EPL tags, with bounded dispatch, retry visibility, and truthful success messaging.
- **Counterpoint transition tooling** — direct Bridge-to-ROS import proof, run-scoped provenance and exceptions, stronger catalog/open-document recovery, and clearer Main Hub operational status.
- **Provider and operational resilience** — hardened Helcim, QBO, IMAP, Shippo, connectivity, backup/update, offline recovery, and transactional outbox boundaries without placing remote provider waits inside customer-money locks.
- **Windows deployment and updates** — rollback-safe Main Hub installation, stronger migration/bootstrap recovery, signed standalone updaters, exact-build deployment selection, and clearer Deployment/Server Manager diagnostics.
- **Release engineering** — Node 24 and Rust 1.91 build tooling, immutable action pins, bounded Dependabot load, per-job Rust caches, concurrent Windows/macOS builds, aggregate Playwright enforcement, and exact-SHA release-candidate promotion.

## Release method

v0.95.0 must use the verified candidate path:

1. Commit the complete v0.95.0 source and documentation with all version manifests in parity.
2. Pass Lint Checks and blocking Playwright on that exact commit.
3. Run the Windows and macOS release workflows with `publish_release=false`.
4. Confirm both candidate runs completed successfully and retained all required signed artifacts.
5. Create `v0.95.0` on that exact candidate commit.
6. Run **Promote verified release candidates** with both run IDs.
7. Verify published updater manifests, build manifests, signatures, deployment ZIP, release title/body, and installed-machine update behavior.

The promotion workflow fails closed when the tag SHA, Windows run SHA, macOS run SHA, GitHub artifact digests, updater build manifests, or deployment package provenance do not agree.
