# Riverside OS v0.85.0 — GO LIVE Readiness: POS Register Hardening

Riverside OS v0.85.0 is the GO LIVE readiness release. It is the result of a systematic end-to-end review of the POS Register (cart, checkout, payments, printing, sessions, offline), Back Office, Settings & Integrations, and Performance. Six critical fixes (A–F) were identified and implemented to harden the register for daily production use.

---

## What Changed

### 1. Fix A — POS Session Token Expiry Pre-Check
* **Early Session Validation**: `useCartCheckout` now probes `GET /api/sessions/current` before tendering. If the session has expired or been closed from another terminal, the cashier gets immediate feedback: *"Your register session is no longer active. Re-open the till to continue."*
* **Prevents Late Rejection**: Previously, a cashier could enter all payment splits and only discover the session was dead at the final server validation. Now the failure happens before any tendering UI is shown.
* **Graceful Degradation**: Network hiccups during the probe are silently ignored — the server will still validate at POST time.

### 2. Fix B — Server-Side Printer Config Persistence
* **Per-Lane Storage**: Printer settings (receipt station IP/port/mode, tag printer, report printer, cash drawer toggle) are now stored in `store_settings.pos_station_config` under a `printer_config.{lane}` key.
* **New API Endpoints**:
  - `GET /api/settings/printer-config/{register_lane}` — read config for a lane
  - `PATCH /api/settings/printer-config/{register_lane}` — write config (admin only)
* **Hydrate on Lane Change**: The Register Overlay calls `hydratePrinterConfigFromServer()` whenever the register lane dropdown changes, restoring the last known settings for that station.
* **Sync on Open**: On successful register open, `syncPrinterConfigToServer()` pushes the current localStorage settings to the server so they follow the lane across devices.
* **Eliminates Drift**: Cashiers no longer need to reconfigure printers when moving to a different terminal or after a browser cache clear.

### 3. Fix C — Blocked Offline Items Recovery UI
* **Live Polling**: The POS cart header polls the offline checkout queue (`localforage`) every 10 seconds via `getCheckoutQueueSummary()`.
* **Amber Syncing Badge**: Shows `${count} syncing` when offline sales are queued and awaiting network recovery.
* **Red Recovery Badge**: Shows `${count} offline sales need recovery` when items are blocked (e.g., stock conflict, server error) and require manual cashier attention.
* **Event-Driven Refresh**: Custom events (`queue_changed`, `print_queue_changed`) trigger immediate re-polling so the UI stays current after checkout or print failures.

### 4. Fix D — Receipt Print Failure Retry Queue
* **Automatic Capture**: When `printRawEscPosBase64()` throws in `ReceiptSummaryModal`, the failed print job (transaction ID, label, base64 payload) is enqueued to `localforage` via `printRetryQueue.ts`.
* **Header Retry Button**: A red danger button (`${count} print retry`) appears in the POS cart header when failures exist.
* **Retry Modal**: Clicking the button opens a modal listing each failed job with:
  - Transaction short ID and attempt count
  - **Retry** button — re-sends the base64 payload to the printer
  - **Dismiss** button — removes the job from the queue
* **Persistence**: The queue survives page reloads and browser restarts.

### 5. Fix E — Dynamic Register Lane Dropdown
* **No More Hardcoded 4 Lanes**: The Register Overlay dropdown was previously hardcoded to 4 options. It now fetches `max_register_lanes` from `GET /api/settings/pos-station-config/public`.
* **Dynamic Generation**: Options are generated with `Array.from({length: maxRegisterLanes}, ...)`.
* **Default 4**: If the setting is absent, the default remains 4 lanes for backward compatibility.
* **Custom Labels**: Lanes 1–4 retain friendly names (Main, iPad, Back Office, Mobile); lanes 5+ are labeled `Station N`.
* **Migration**: `058_pos_station_config.sql` adds the `pos_station_config JSONB` column to `store_settings`.

### 6. Fix F — Helcim Terminal Stream Auto-Reconnect
* **Fallback Polling**: `NexoCheckoutDrawer` now runs a `setInterval` fallback poll (every 4 seconds) alongside the SSE stream.
* **Silent SSE Recovery**: If the SSE connection drops without firing `onerror`, the poll continues refreshing the terminal attempt status via `refreshHelcimAttempt()`.
* **Clean Cancellation**: The interval is properly cleared in the useEffect cleanup, and the `stopped` flag prevents race conditions after component unmount.
* **Prevents Stuck Terminals**: Previously, a silent SSE disconnect could leave the payment modal spinning indefinitely. Now the poll will eventually surface a terminal result or timeout.

## Hotfix — May 28, 2026

Four additional fixes and one seed improvement shipped after the initial v0.85.0 tag.

### Receipt Line Wrapping
* **Root cause**: The `receiptline` library wraps the left column of a two-column row (`name | price`) independently. Product names like "RMS CHARGE PAYMENT" (22 chars) caused a single trailing letter to overflow onto a second line.
* **Fix**: `clip_item_name(name, 28)` in `receipt_escpos.rs` hard-clips at the nearest word boundary before embedding the name in the receiptline format. All item names now render on a single line at CPL 42.

### Alteration Ticket # Save Error
* **Root cause**: The Ticket # input called `updateAlteration` (a `PATCH` API call) on every keystroke, rapidly firing concurrent requests that caused server-side race errors.
* **Fix**: Local `ticketDraft` state; save fires only on `onBlur` or Enter, and only when the value actually changed.

### Alteration Scheduling UX Refresh
* **Removed**: `RosieInsightSummary` / "ROSIE INSIGHT" label from the Capacity Outlook panel. AI still powers slot suggestions, but the panel is now brand-neutral.
* **Renamed**: Scheduling buttons are now **Auto-Schedule** (AI-powered, purple) and **Manual Schedule** (neutral). Tabs relabeled to **1. Work Items** / **2. Schedule**.
* **E2E**: `alterations-smart-scheduler.spec.ts` updated to match the new tab label and removed Rosie-panel assertions.

### Gift Card Double-Input Bug
* **Root cause**: The global `window keydown` listener fired `appendAmountKey` on `0–9`, and clicking a numpad button fired `onClick → appendAmountKey` simultaneously, doubling every digit (pressing `6` produced `66`).
* **Fix**: Guard in the keydown handler — skip when `document.activeElement` is a `<button>` (numpad handled by `onClick`) or the card code `<input>` (handled by React `onChange`).

### Custom Order Operational SKUs
* Seeded `Custom Orders` product category and four operational SKUs (`100` HSM Suit, `105` HSM Sport Coat, `110` HSM Slacks, `200` Individualized Shirt) matching `CUSTOM_ORDER_SUBTYPES` in `customOrders.ts`. Applied via `seed_core_required.sql`.

## Hotfix — May 31, 2026

Additional scroll alignment adjustments, support configuration consolidation, and standalone macOS DevOps improvements have been deployed:

## Countersync Desktop Sync Bridge GUI App (NEW)

Riverside OS v0.85.0 now includes the native **Riverside Countersync** desktop application for Windows (`deployment/counterpoint-bridge-gui`). It wraps the background Node SQL extraction engine into a clean, zero-config GUI.

* **Single Standalone Installer**: The entire sync bridge—including background extraction scripts and dependency managers—is compiled and distributed as a single MSI/EXE installer.
* **Tauri Resource Bundling**: Script resources are packed directly into the application bundle, checking and resolving resource paths dynamically.
* **Automated Background Dependency Setup**: Automatically detects missing dependencies (`node_modules`) on launch and executes `npm install` cleanly in the background.
* **Live Process Log Console**: Streams real-time stdout and stderr from the background child extraction engine, color-coding successes, heartbeats, and database connectivity errors.
* **Interactive SQL Query Tester**: Exposes a safe environment to test configured sync mappings or custom raw SQL queries, returning a 10-row visual table preview before running exports.
* **Visual Status Control**: Start, stop, dry-run, and manual schema extraction runs controlled dynamically via a modern WowDash glassmorphism dashboard.

## Migration

Run before deploying:

```sql
-- 058_pos_station_config.sql
ALTER TABLE store_settings ADD COLUMN IF NOT EXISTS pos_station_config JSONB DEFAULT '{}'::jsonb NOT NULL;
```

The server auto-applies this on startup via `run_migrations()`. For manual application:

```bash
export DATABASE_URL="postgresql://..."
./scripts/apply-migrations-psql.sh
```
