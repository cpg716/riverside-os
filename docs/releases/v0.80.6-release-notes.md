# Riverside OS v0.80.6 — Production Hardening: In-App Host Server Updater, Help Viewer Authentication Fallback, and Rosie Insights Timeout Extension

Riverside OS v0.80.6 is a production hardening release focused on improving release deployment capabilities and resolving UI/network timeouts on older workstation hardware. It introduces a robust, self-contained in-app server updater, adds authentication fallbacks to the staff help viewer, and extends Rosie Insights timeouts.

---

## What Changed

### 1. Robust In-App Host Server Updater
* **Tauri-Powered Server Updates**: Implemented a native server update orchestration layer in Tauri (`server_updater.rs`).
* **Elevated PowerShell Execution**: The desktop app can now download the latest release bundle, unpack the binaries and installer scripts, and execute the installation script under Windows Admin privileges via a standard UAC dialog.
* **Update Manager UI Dashboard**: Enhanced `UpdateManagerPanel.tsx` and `appUpdater.ts` to display server status, download progress, and step-by-step update tracking (*Downloading package...*, *Waiting for Admin authorization...*, *Applying migrations...*, *Restarting services...*), allowing for a 100% scriptless, self-contained update workflow from the running app.

### 2. Staff Help Viewer Authentication Fallback
* **Session Auth Recovery**: Hardened `server/src/api/help.rs` and help routing. If active staff profile headers are absent or incomplete during help index retrieval, the endpoint gracefully falls back to verification of the active POS register session credentials.
* **Zero-Friction Help Search**: Resolves search errors and content loads blocking cashier stations during guest-level lookups and shifts transitions.

### 3. ROSIE Insights Optional Timeout Expansion
* **Client-Side Timeout Extension**: Increased the local `ROSIE_OPTIONAL_INSIGHT_TIMEOUT_MS` parameter in `client/src/lib/rosie.ts` from 15 seconds to 120 seconds.
* **Low-Power Hardware Support**: Extends the client-side timeout window to ensure slower CPU-only hosts (such as Intel Macs or older Server PCs running Gemma 4 E4B models without dedicated GPUs) can finish generating complex operational insights without triggering early network aborts.

---

### 4. Helcim Card-Token Purchase Transaction Safety
* **Atomic Success Path**: `process_helcim_card_token_purchase` (`server/src/api/payments.rs`) now runs the `payment_provider_attempts` status UPDATE inside the same database transaction that locks the register session. Previously the transaction was started but the UPDATE executed against `&state.db`, leaving the attempt stuck `pending` if the process crashed between the Helcim API success and the local DB commit.

### 5. QBO Duplicate-Journal Prevention
* **Syncing State Lock**: The QBO sync job (`server/src/jobs/qbo_sync.rs`) transitions approved staging rows to `syncing` before POSTing to QuickBooks. If the local DB update fails after QBO succeeds, the row stays `syncing` (or `failed`) and can never revert to `approved`, eliminating duplicate journal entries after the 24-hour QBO `requestid` deduplication window expires.

### 6. RMS Charge Double-Reversal Guard
* **Resolution Status Check**: `reverse_rms_record_manual` (`server/src/api/pos.rs`) now queries `resolution_status` before applying a refund or reversal. If the record is already `refunded` or `reversed`, the request is rejected with a clear error instead of silently overwriting metadata.
* **Dedicated Permission Key**: The `reverse_rms_charge_purchase` and `reverse_rms_charge_payment` endpoints now require the explicit `customers.rms_charge.reverse` permission. Previously they accepted `orders.refund_process` **OR** `customers.rms_charge.manage_links`, which was overly broad.

### 7. Offline Queue Flush Timeout
* **AbortController Guard**: `flushCheckoutQueue` (`client/src/lib/offlineQueue.ts`) now attaches a 15-second `AbortController` timeout to every replay `fetch`. This prevents half-open TCP connections from hanging the flush loop indefinitely and blocking subsequent queued items.

### 8. Backorder Failure Notification
* **Procurement Alert**: When `create_backorder_from_short_lines` fails after PO receipt (`server/src/api/purchase_orders.rs`), an app notification is now emitted to all staff with `procurement.mutate` permission. Previously the failure was only logged and could go unnoticed.

---

## Deployment Notes

* **Workspace Version Parity**: All build manifests (`package.json`, Cargo workspaces, Tauri configurations, workflows, and deployment templates) are locked at version `0.80.6`.
* **Database Updates**: Database migrations are automatically checked and applied on server launch.
* **Permission Migration**: Existing staff with `customers.rms_charge.reverse` already retain access; staff who previously relied on the broad `orders.refund_process` fallback for RMS reversals must be explicitly granted `customers.rms_charge.reverse` in **Settings → Staff access defaults** or per-staff profiles.
