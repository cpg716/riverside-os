## 2026-07-22 replacement candidate

The replacement candidate resets every sale-local checkout drawer and Card Not Present handoff at a completed or cleared sale boundary. It accepts a hosted approval only for the exact request nonce, provider attempt, checkout, customer, and amount. **Clear Sale** releases local Register state while preserving processor evidence in Payments Health, and an informational review note cannot hide an approval that remains unlinked. Local unconfirmed recovery evidence clears only from an exact Main Hub resolution that names the checkout, recovery job, and completed Transaction Record. The new **Match Existing Paid Transaction** workflow verifies immutable transaction/payment facts and closes only the recovery record; it creates or moves no money.

Local Playwright is intentionally skipped for this replacement at operator direction. Non-E2E source checks, the guarded pre-retag path, exact-commit GitHub Lint and blocking Playwright, signed packaging, updater/build-manifest verification, and physical Main Hub/terminal verification remain required. The historical publication evidence below does not certify this candidate.

## Production update incident

On 2026-07-13, the published `86724e73` Windows package failed during a Main Hub update. The pre-migration dump used the limited `riverside_app` account and could not read the production-only `ros_repair_backup` schema. Rollback then restored an environment file whose application password no longer matched the PostgreSQL role, leaving the API offline until `DATABASE_URL` was synchronized from the protected deployment configuration. The restored v0.90.0 API returned HTTP 200 and the database remained intact.

The original `86724e73`, superseded `c3826bf6`, and prior `94f962c` Windows packages must not be reused. The last verified pre-candidate build `446413bf` passed its exact-commit GitHub and release workflows. Its update package explicitly preserves installed ROSIE LLM/STT/TTS assets, environment settings, processes, and scheduled task instead of invoking ROSIE setup. A successful Main Hub update/readiness check remains required for operational signoff.
