# v0.90.0 READY TO TEST Blocker Audit

Date: 2026-06-06
Scope: blocker-focused audit for the Riverside OS v0.90.0 store test run.
Mode: AUDIT, with only small targeted fixes for confirmed blockers.

## 1. Executive Summary

Recommendation: **GO FOR READY-TO-TEST** for the v0.90.0 store test run.

The core POS, customer order, wedding, QBO, and receiving paths have real wiring and E2E coverage, but this audit found concrete blockers in high-risk Wedding Manager and alterations flows. The targeted blockers were fixed in a targeted way:

- Wedding group pay ledger rows were returning a transaction id in the `wedding_member_id` field, so Wedding Manager financial context could not reliably attribute a group payment to the correct member.
- Wedding financial context could return HTTP 500 because the reporting view returned `free_suits_marked` as `INT8` while the Rust tuple expected `i32`.
- Completing an alteration attempted to update `transaction_lines.updated_at`, but `transaction_lines` has no `updated_at` column.

The targeted top-10 proof set now passes across wedding group pay, wedding readiness, pickup certification, QBO staging/audit, receiving API/UI, server checks, frontend typecheck, frontend lint, and Help impact. This is a GO for the v0.90.0 store test run, not a production release publish signoff; the manual store checklist below still needs to be executed against the exact build the store will test.

## 4. Findings By Workflow

### 4.1 POS Sale Completion And Checkout

The POS checkout flow has strong pre-finalization checks. `useCartCheckout` probes `GET /api/sessions/current` before tendering, requires a POS token, blocks offline shipping/order payments, validates deposit allocation, submits `checkout_client_id`, and sends payment splits, order payments, wedding disbursements, alteration intake, shipping quote, tax, and exemption fields. `NexoCheckoutDrawer` prevents finalization until the drawer is balanced and required operator/tax-note/check fields are present. The backend `CheckoutRequest` shape matches the client payload, rejects checks without `check_number`, rejects unapproved Helcim payments, and uses idempotency.

Residual risk: the client still retries a 5xx checkout failure once after a one-second delay. That was pre-existing and was not changed during this audit.

### 4.2 Special, Custom, And Wedding Order Creation

The checkout and order-contract tests cover custom, special, and wedding-linked fulfillment behavior. Transaction detail/list paths expose wedding summaries and order-scope filtering. Order records are not a separate ambiguous "order" table in the reviewed paths; the financial transaction and fulfillment order semantics remain distinct.

No blocker was found in the reviewed order classification wiring.

### 4.3 Wedding Manager Package/Member Item Loading Into ROS/POS

Wedding purchase context loads linked member and party suit variants. POS cart shows a wedding checklist with Take now, Order, and Measure actions for linked sellable purchase items. Unlinked wedding checklist items are visible with "Not linked to a sellable product yet."

Watch item: this appears ready for linked suit/product variants, but the audit did not add a broader package-to-product model. If the store expects package assignment to automatically populate multiple products per member, that should be tested explicitly during the manual store run.

### 4.4 Wedding Deposits, Split Deposits, Group Pay, And Balances

Confirmed blocker fixed: group payout ledger rows were returning `payment_allocations.target_transaction_id` as `wedding_member_id`. That made Wedding Manager financial context report a transaction id where member attribution was required. The query now joins the target transaction's wedding member and returns `wm.id`.

Confirmed blocker fixed: the same financial-context endpoint returned HTTP 500 after group pay in the targeted E2E scenario because `reporting.wedding_party_economics.free_suits_marked` decoded as `INT8` into an `i32` tuple slot. The query now casts `free_suits_marked::int` to preserve the existing API shape.

The targeted Playwright contract now asserts that a $50 group payout appears in party financial context and is attributed to the attached wedding member id.

### 4.5 Wedding Lifecycle Status Propagation

Readiness logic consumes transaction-line lifecycle counts, member fields, alterations, received status, and pickup state. Order lifecycle guards prevent ready-for-pickup before received status and alteration readiness unless a Manager Access override is supplied with justification.

No additional blocker was fixed in this area beyond the alteration-ready propagation bug below.

### 4.6 Alterations, Fittings, And Pickup Readiness

Confirmed blocker fixed: when an alteration moved to ready status, `patch_alteration` attempted to set `transaction_lines.alteration_ready = true, updated_at = now()`. The `transaction_lines` schema does not define `updated_at`, so a completed alteration could fail to propagate readiness back to the linked line. The update now only sets `alteration_ready = true`.

The pickup certification contract was rerun after the readiness fix and passed. The fixture now uses the same Manager Access readiness override path enforced by the API instead of simulating an impossible manual received transition.

### 4.7 Customer Profile, Transaction Record, And Order Visibility

Orders workspace uses transaction list/detail APIs with order scope and audit/detail loading. Customer Relationship Hub loads transaction history, order-scope records, alterations, deposits, store credit, and lifecycle surfaces. Wedding transaction detail exposes wedding summary data.

No blocker was found in the reviewed visibility paths.

### 4.8 API/Data Contract, Migrations, Seeded Data, And Permissions

Routes for inventory, alterations, transactions, products, procurement imports, purchase orders, QBO, sessions, customers, order lifecycle, and weddings are mounted. The reviewed endpoints use Staff Access/Manager Access gates rather than bypassing permissions.

No current blocker was found in the reviewed route mounting, seeded Staff Access/Manager Access paths, or targeted E2E contract runs.

### 4.9 QBO/Accounting Readiness

QBO workspace loads granular mappings, mappings, credentials, staging, and accounts cache. Backend actions gate view, mapping edit, staging approve, sync, retry, revert, void, and drilldown operations with QBO permissions. Journal logic warns on missing mappings rather than silently inventing fallback accounts.

The targeted QBO staging/audit specs were rerun and passed. The journal-balance contract was tightened so the test proves the transactions it created instead of assuming no other same-day rows exist in a persistent E2E database.

### 4.10 Product Hub, Procurement, POs, Receiving, And Inventory Availability

PO receiving has physical inventory lockout, idempotent receipt replay, row-level variant locking, WAC update from invoice unit cost, separate freight landed-cost component handling, inventory transaction audit rows, special/custom/wedding reserved-stock allocation, PO line receipt updates, and wedding-member received propagation. Procurement import is review-first, with unmatched/new lines staying in review and conversion creating PO/direct invoice records rather than silently posting stock.

Physical Inventory work was completed outside this audit. After that, the receiving API and UI contracts were rerun and passed for the Product Hub/procurement/PO receiving proof set.

## 5. Confirmed Blockers

1. **Wedding group pay member attribution broken in financial context.**
   - Impact: Wedding Manager could show or process group payment rows with a transaction id in `wedding_member_id`.
   - Fix: return the target transaction's `wedding_members.id` in the group payout ledger union.

2. **Wedding financial context 500 after group pay.**
   - Impact: Wedding Manager balance context could fail to load during a core wedding deposit/group-pay test flow.
   - Fix: cast `free_suits_marked::int` in the economics query to match the Rust API tuple.

3. **Alteration-ready propagation could fail on nonexistent column.**
   - Impact: completed alterations could fail to mark linked transaction lines ready, blocking pickup readiness.
   - Fix: remove `updated_at = now()` from the `transaction_lines` update.
