================================================================================
  Riverside OS — Counterpoint → Riverside import bridge (Windows folder)
================================================================================

What this is
  A tiny Node.js program on the Counterpoint / SQL Server PC that reads
  Counterpoint data and pushes it to Riverside OS on your network.

What you need on this PC (Windows 10 is supported)
  1. Node.js 18+ (LTS) from https://nodejs.org/  — install with defaults
     (64-bit Windows installer is fine on Windows 10).
  2. This folder (unzipped), placed anywhere you like, e.g.:
       C:\Riverside\counterpoint-bridge\

First full Counterpoint load (avoid “nothing imported” days)
  • On Riverside: Settings → Integrations → Counterpoint → enable ROS staging when you want staff review
    before live writes. Staging ON queues every batch until staff Apply or Discard it from the command center.
  • Apply ROS DB migrations through at least 95 (staging/GUI) on the Postgres that the API uses.
  • Bridge: ROS_BASE_URL = LAN IP:3000 of the Main Hub ROS API PC.
  • Runtime schema mapping: keep .env to ROS_BASE_URL, COUNTERPOINT_BRIDGE_TARGET_MODE=ros_import_first,
    and SQL_CONNECTION_STRING; the v8.4 bridge probes schema after SQL connect.

First-time setup
  0. (Recommended) After you set SQL_CONNECTION_STRING in .env, double-click DISCOVER_SCHEMA.cmd.
     It probes your Counterpoint database and tells you exactly which columns/tables differ from the
     template (cost field, points, gift/loyalty tables, etc.). No Riverside server or token required.
     A copy of the output is saved as counterpoint-schema-report.txt in the same folder (set
     CP_DISCOVER_OUTPUT in .env to change path, or CP_DISCOVER_OUTPUT=0 to skip the file).

  1. Double-click  START_BRIDGE.cmd
     - It will run "npm install" once.
     - It will create .env if missing (from .env.example, or from env.example if packed that way) — you must edit it.

  2. Open .env in Notepad and set at least:
       SQL_CONNECTION_STRING=...   (your Counterpoint SQL Server connection)
       ROS_BASE_URL=http://IP:3000 (IP of the PC running Riverside OS)
       COUNTERPOINT_BRIDGE_TARGET_MODE=ros_import_first

     Do not add legacy CP_*_QUERY lines for normal setup. Run DISCOVER_SCHEMA.cmd or GUI Auto Config
     if SQL reports a table/column problem.

  3. Double-click START_BRIDGE.cmd again. The bridge defaults to one full import pass, then exits.

Same network
  Use the ROS computer's LAN IP, e.g. ROS_BASE_URL=http://192.168.1.100:3000
  (Ping that PC from this one to confirm connectivity.)

Docker (this Riverside OS repo)
  docker-compose.yml runs PostgreSQL (host port 5433) and optional Meilisearch (7700).
  It does NOT run the Rust API in a container — the API still runs on a host process
  (e.g. npm run dev / release binary) and listens on port 3000 by default.
  Point ROS_BASE_URL at wherever that HTTP API is reachable (same machine: host LAN IP
  + :3000, not the Postgres port). DATABASE_URL on the server uses localhost:5433 when
  Postgres is the Compose db.

Runtime mapping scope
  - Staff/users, sales reps, buyers
  - Vendor profiles and vendor item links
  - Customers, loyalty point balances, customer notes, and store-credit balances when visible
  - Catalog products, matrix variants, barcodes, prices, costs, and categories
  - Inventory quantities
  - Gift-card current balances when visible
  - Historical tickets, lines, payments, and notes
  - Open PS_DOC orders when visible
  - Receiving history when visible

  Import order is fixed in the bridge, not by rearranging .env: staff → vendors → customers →
  store credit opening → notes → category_masters → catalog → inventory → vendor_items → gift_cards →
  tickets → optional open documents → loyalty. The v8.4 bridge probes the live company database and
  builds runtime mappings from INFORMATION_SCHEMA before each run.

  Staff runs before customers and tickets so rep/cashier attribution resolves.

  See .env.example for the three connection settings.
  Full guide: docs/COUNTERPOINT_SYNC_GUIDE.md in the Riverside OS repo.

  Runtime schema mapping: keep .env to ROS_BASE_URL, COUNTERPOINT_BRIDGE_TARGET_MODE=ros_import_first, and SQL_CONNECTION_STRING.
  The GUI Auto Config action, DISCOVER_SCHEMA.cmd, or `node index.mjs auto-config` confirms which live
  v8.4 tables and columns the bridge will use. Legacy CP_*_QUERY entries are ignored unless
  CP_SQL_ENV_OVERRIDES=1 is explicitly set for expert recovery work.

  Staging ingest: when “Counterpoint staging” is enabled in Riverside Settings → Integrations → Counterpoint, GET
  /api/sync/counterpoint/health returns counterpoint_staging_enabled=true and the bridge POSTs each entity payload to
  /api/sync/counterpoint/staging instead of applying immediately; staff Apply or Discard batches in the Counterpoint
  hub Inbound queue tab.

Troubleshooting
  - Connection refused to ROS → firewall on ROS PC must allow port 3000 from this PC.
  - health failure on ROS → confirm ROS_BASE_URL points to the Main Hub ROS API on port 3000 and that /api/sync/counterpoint/health returns JSON.
  - "invalid object name …" on customer sync → set Database= in SQL_CONNECTION_STRING to your
    Counterpoint company DB (same as in SSMS). Then run GUI Auto Config or DISCOVER_SCHEMA.cmd so the
    bridge can rebuild mappings from the live schema.
  - "Invalid column name 'STAT'" → not from ticket SQL. Run GUI Auto Config or DISCOVER_SCHEMA.cmd; the
    bridge should omit unavailable staff/catalog columns from its runtime SQL.
  - Other "Invalid column name" / "Invalid object name" on sync → your Counterpoint SQL schema
    differs from the expected v8.4 shape. Run GUI Auto Config or DISCOVER_SCHEMA.cmd and review the
    generated report. Only use CP_SQL_ENV_OVERRIDES=1 with manual CP_*_QUERY values as an expert fallback.
  - Customer notes skipped_no_customer → run customer sync successfully first (notes attach by cust_no).
  - ROS error duplicate key vendors_name_key → fixed in current ROS server: second vendor with the same CP name imports as "Name [VEND_NO]". Update/restart the API if you still see this.
  - Reading the log: [staff]/[customers] SQL errors block those imports; then notes show skipped_no_customer
    because cust_no is not in ROS yet. Run Auto Config/discovery for the failing entity before retrying.
  - Inventory batches show skipped=200, updated=0 or 2 → ROS only updates rows that match an existing
    variant SKU or counterpoint_item_key. Catalog must run successfully (bridge runs catalog before
    inventory each cycle). Wrong location mapping can also cause skips.
  - "Invalid column name 'DOC_TYP'" (vendors, inventory, catalog subqueries, tickets) → your PS_TKT_HIST
    may not use that column name. Run Auto Config/discovery so the bridge can build date/activity filters
    against the columns that actually exist. Then re-run START_BRIDGE.cmd.
  - "Invalid object name 'IM_CATEG'" → run Auto Config/discovery. The runtime mapper can fall back to
    category values present on IM_ITEM.
  - "Invalid column name 'IS_GRD'" → run Auto Config/discovery. The runtime mapper derives grid status from
    IM_INV_CELL when the item table does not expose a grid flag.
  - "Invalid column name 'MERCH_CR_BAL'" → run Auto Config/discovery. Store-credit opening balances only run
    when a supported balance source is visible.
  - "Incorrect syntax near ORDER" on customers → likely a stale expert CP_*_QUERY override. Remove the
    manual override or rerun Auto Config/discovery.
